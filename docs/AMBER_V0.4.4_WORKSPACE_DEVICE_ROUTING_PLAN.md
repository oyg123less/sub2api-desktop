# Amber v0.4.4 工作区隔离、设备定向共享与官网开发计划

> 文档状态：待确认，尚未开始业务代码开发
> 目标版本：v0.4.4
> 基线版本：v0.4.3
> 修订日期：2026-07-20
> 发布策略：只发布一个 v0.4.4 正式安装包，不向普通用户分发中间构建
> 核心主题：移除好友前置流程、同机多用户彻底隔离、同账号多设备定向回流、一键共享、可靠注入
> 本期不讨论：Cloudflare 容量规划与计费扩容

---

## 0. 执行结论

v0.4.4 不是在现有好友页面上继续叠加按钮，而是重新确立三条产品边界：

1. 云账号只负责身份、同步和授权，不代表某一台物理设备。
2. 本地工作区必须属于唯一云用户，不同云用户不得共用账号、代理、同步队列或共享密钥。
3. 需要本地回流的共享必须指向具体设备，不能仅按云用户选择任意在线设备。

最终普通用户看到的共享流程为：

```text
共享者：选择账号 -> 开始共享 -> 发送共享码和临时密码
接收者：输入共享码和临时密码 -> 连接并使用
```

好友申请、等待接受、好友列表、选择好友等步骤全部退出主流程。连接成功后产生的是“授权记录”，不是社交好友关系。

本版本必须一次完成以下交付：

- 同机多云账号独立工作区。
- 多设备同账号的共享设备定向路由。
- 好友功能退出产品界面与共享前置条件。
- 共享池故障隔离与安全故障转移。
- Codex“启动服务并注入”闭环。
- 云同步网络诊断和入口切换。
- v0.4.4 官网、下载页、文档页和常见问题页。
- v0.4.3 数据无损迁移、回滚保护和完整测试。

## 1. 当前两个最高风险问题

### 1.1 同一设备切换不同云用户会污染数据

当前本地 `cloud_session` 是单例，但下列数据没有按云用户分区：

- `accounts`
- `proxies`
- `codex_remote_targets`
- `cloud_sync_settings`
- `cloud_sync_outbox`
- `cloud_sync_tombstones`
- 本地设置和部分同步状态

现有流程可能出现：

```text
用户 A 登录并同步
-> 退出 A
-> 同一数据库登录用户 B
-> B 的云数据合并进 A 的本地数据
-> 修改原属于 A 的账号
-> 该账号被作为 B 的脏数据上传
```

更严重的情况是 A 遗留的 outbox 已使用 A 的保险库密钥加密，B 登录后仍可能尝试使用 B 的会话提交该批次，造成持续同步失败或错误归属。

此问题不能仅通过“退出时清空同步游标”修复。同步游标、待发送批次、删除墓碑、普通账号、代理和设置必须同时隔离。

### 1.2 同一云账号多设备在线时回流目标不确定

当前 Worker 的 Owner Relay 逻辑优先选择在线主设备，主设备离线后选择其他在线设备，但没有确认目标设备是否：

- 拥有请求中的 `account_uid`。
- 已启用该账号。
- 具备对应代理。
- 能通过该设备当前网络出口访问上游。
- 处于健康状态。

因此可能发生：

```text
账号只存在于设备 B
设备 A 是主设备且在线
-> Worker 把请求发给 A
-> A 返回 no_account
-> 请求失败，Worker 不会改走 B
```

即使账号同步到了 A、B 两台设备，代理也可能是 `127.0.0.1` 本地端口，只在其中一台设备有效。单纯按“主设备”路由不能保证可用性，也可能改变共享者期望的 TLS、IP 和网络出口。

## 2. 远程共享软件的可借鉴原则

### 2.1 网易 UU 远程

官方远程协助流程使用设备 ID 与验证码。所谓“家人/好友”和“伙伴设备 ID”是使用场景文案，实际并不要求双方先建立社交好友关系。近期连接通过“最近连接”或设备 ID 下拉记录提供。

可借鉴内容：

- 连接码代表目标设备或连接端点。
- 临时密码作为第二个验证因素。
- 最近连接用于重复使用，不需要好友申请。
- 同账号下的多台设备仍然是不同设备。

参考：<https://uuyc.163.com/features/support/>

### 2.2 ToDesk

ToDesk 个人版核心是设备列表、设备代码和最近连接。团队版增加成员管理、设备管理和连接管控，但“成员”是权限主体，不是聊天好友。

可借鉴内容：

- 设备是路由目标，账号是管理身份。
- 团队成员和设备授权分开建模。
- 管理者可以查看连接记录、收回设备权限和中断连接。

参考：<https://www.todesk.com/helpcenter/solo.html>

### 2.3 向日葵

向日葵个人版使用识别码、验证码、绑定设备和设备列表；企业版使用成员、角色和设备权限。连接始终面向目标设备，而不是随机选择登录了同一账号的设备。

可借鉴内容：

- 长期绑定和临时协助是两种模式。
- 目标设备身份稳定，验证码可以更新。
- 权限管理属于设备或组织，不依赖社交好友。

参考：<https://service.oray.com/product/sunlogin.html>

### 2.4 Amber 应采用的对应关系

| 远程软件概念 | Amber 对应概念 |
| --- | --- |
| 设备 ID | Amber 设备 ID / 共享承载设备 |
| 设备验证码 | 临时共享密码 |
| 最近连接 | 最近共享 |
| 绑定设备 | 已接收共享 |
| 团队成员权限 | 接收者独立授权和 Guest Key |
| 设备离线 | 共享承载设备离线 |
| 备用设备 | 用户显式配置的回流备用设备 |

Amber 不应复制好友系统，而应复制“设备定向、短凭据连接、长期权限独立管理”的原则。

## 3. 不可妥协的数据与路由约束

以下约束必须写入代码测试，不能只依赖界面提示：

### 3.1 工作区约束

1. 一个本地工作区最多绑定一个 `cloud_user_id`。
2. 绑定后的工作区退出登录也不会解除归属。
3. 不同云用户不得共用 SQLite、加密密钥、outbox、tombstone 和同步游标。
4. 一个 Amber 进程同一时间只允许一个活动工作区和一个活动云会话。
5. 切换工作区必须先停止同步、Owner Relay 和新请求接入，再切换数据库。
6. 登录返回的 `user_id` 与工作区归属不一致时，必须在保存会话前失败。
7. outbox、tombstone 和冲突记录仍需保存 `owner_user_id`，形成数据库隔离之外的第二道防线。
8. 任何归属不明的旧 outbox 都不得自动上传。

### 3.2 设备路由约束

1. 云用户身份不能直接作为本地回流目标。
2. 每个本地回流共享必须记录明确的 `host_device_id`。
3. 新共享默认绑定创建共享的当前设备。
4. 设备 B 登录同一云账号不能自动接管设备 A 创建的共享。
5. 只有用户显式添加的备用设备才具备接管资格。
6. 备用设备必须上报它拥有目标 `account_uid` 且健康检查通过。
7. 上游请求开始前可以故障转移，上游请求开始后禁止重放。
8. 无合格设备时返回明确的设备离线错误，不得随机选择任意在线设备。

### 3.3 授权约束

1. 接收共享不需要好友关系。
2. 每位接收者获得独立授权和独立 Guest Key。
3. 一位接收者被暂停或删除不能影响其他接收者。
4. 共享码和临时密码只能领取授权，不能直接调用模型。
5. 好友历史数据不得在升级过程中被直接删除。

## 4. 同一设备多云用户的完整修复

### 4.1 采用独立工作区，不在所有业务表硬塞 user_id

推荐目录结构：

```text
AmberData/
├── workspace-index.json
├── legacy/                       # 原 v0.4.3 数据位置或只读恢复入口
└── workspaces/
    ├── ws_<随机ID>/
    │   ├── sub2api.db
    │   ├── key
    │   ├── codex_known_hosts
    │   └── profile.json
    └── ws_<随机ID>/
        ├── sub2api.db
        ├── key
        ├── codex_known_hosts
        └── profile.json
```

每个工作区独立拥有：

- 普通账号与凭据。
- 代理及代理密码。
- Codex 远程目标与主机密钥。
- 云登录会话和保险库密钥。
- 同步游标、outbox、tombstone、冲突记录。
- 收到的共享和 Guest Key。
- 共享承载设备身份。
- 本地设置、请求日志和运行状态。

`workspace-index.json` 只保存工作区 ID、目录、脱敏显示名和活动工作区，不保存 Token、密码、Guest Key 或私钥。

### 4.2 首次升级迁移

v0.4.4 第一次运行时执行：

1. 在不启动同步和 Relay 的状态下打开旧数据库。
2. 使用 SQLite 备份 API 创建迁移前快照，不直接复制正在使用的 WAL 文件。
3. 检查 `cloud_session`、`cloud_identities`、`cloud_received_keys`、`cloud_connect_host_state` 中出现的云用户 ID。
4. 只有一个明确用户时，将旧工作区绑定给该用户。
5. 当前存在登录会话时，以会话 `user_id` 为首要归属证据。
6. 没有会话但只有一个历史用户时，要求用户确认后绑定。
7. 出现多个历史用户、归属不明 outbox 或异常元数据时，进入“旧数据恢复工作区”。
8. 恢复工作区默认关闭云同步和回流，绝不猜测数据属于谁。
9. 用户登录对应云账号后创建干净工作区，从该账号云保险库拉取数据。
10. 用户可以从恢复工作区显式选择账号和代理导入，禁止后台自动合并。

迁移不得：

- 删除原数据库。
- 清空旧账号。
- 自动把全部账号复制到每个云用户。
- 把归属不明 outbox 发送给当前登录用户。
- 因迁移失败覆盖原始文件。

### 4.3 登录、退出和切换规则

#### 登录同一用户

```text
打开已绑定工作区
-> 验证登录返回 user_id
-> 与 workspace_meta.bound_cloud_user_id 一致
-> 保存会话
-> 启动同步与 Relay
```

#### 登录不同用户

```text
当前工作区已绑定 A
-> 登录响应为 B
-> 在保存 session 前返回 workspace_account_mismatch
-> 提示切换或创建 B 的工作区
-> 绝不覆盖 A 的 cloud_session
```

#### 退出登录

退出顺序必须是：

1. 停止新的同步任务。
2. 关闭该用户 Owner Relay。
3. 尝试撤销远端 refresh token。
4. 即使远端注销因网络失败，也必须完成本地会话清理。
5. 清空内存 access token、vault key 和重试状态。
6. 保留工作区归属和本地数据。
7. 不删除共享配置、账号、代理或日志。

#### 切换工作区

1. 检查当前设备是否承载活动共享。
2. 如果切换会导致共享离线，明确显示影响数量。
3. 停止接收新请求。
4. 给进行中的本地请求最多 30 秒完成时间。
5. 关闭 API Server、同步、Relay 和数据库。
6. 原子更新活动工作区索引。
7. 使用新 `--data-dir` 重启 Sidecar。
8. 恢复该工作区自己的本地服务设置。
9. 前端重新获取完整状态，不复用上一个用户的内存缓存。

### 4.4 工作区界面

云账户头像菜单增加：

- 当前工作区邮箱。
- 切换账号。
- 添加云账号。
- 本地恢复工作区。
- 工作区管理。

普通页面不展示数据库路径、workspace ID 或同步内部字段。

首次把已有本地数据绑定给云账号时必须明确提示：

```text
这个工作区包含 N 个账号和 M 个代理。
绑定后，它们只会与当前云账号同步，不会与其他云账号共用。
```

### 4.5 防御性数据库字段

即使每个工作区使用独立数据库，也应增加：

- `workspace_meta.bound_cloud_user_id`
- `cloud_sync_outbox.owner_user_id`
- `cloud_sync_tombstones.owner_user_id`
- `cloud_sync_conflicts.owner_user_id`
- `cloud_sync_runtime.owner_user_id`

发送同步批次前执行硬校验：

```text
workspace owner == session user == outbox owner
```

任一不一致时停止同步并将批次隔离，错误码使用 `cloud_workspace_owner_mismatch`。

## 5. 同一云账号多设备的共享路由修复

### 5.1 设备是共享承载目标

每台 Amber 安装实例保持唯一 `device_public_id`。设备名称由用户设置，例如：

- 家里电脑
- 办公室电脑
- 笔记本

创建需要本地回流的共享时，默认写入：

```text
host_device_id = 当前设备
route_policy = fixed_device
```

共享码解析到“共享端点”，共享端点再解析到具体承载设备。不能再直接使用 `owner_id` 选择任意连接。

### 5.2 两种路由模式

#### 固定设备，默认模式

- 共享始终走创建共享的设备。
- 保持该设备的代理、TLS、IP 和账号环境。
- 设备离线时显示“共享设备离线”。
- 不自动切换其他设备。

该模式最接近 UU 远程、ToDesk 和向日葵通过设备码定位目标设备的行为，也是普通用户默认模式。

#### 主备设备，高级模式

共享者可以显式添加备用设备并排列优先级：

```text
主设备：办公室电脑
备用 1：家里电脑
备用 2：笔记本
```

备用设备必须同时满足：

- 在线且心跳未超过 60 秒。
- 本地存在目标 `account_uid`。
- 账号启用。
- 账号未处于限额或冷却状态。
- 所需代理在该设备上真实可用。
- 当前并发未满。

只有主设备在上游请求开始前不可用时才允许切换备用设备。

### 5.3 Worker Relay v2 设备库存协议

设备连接 WebSocket 后发送：

```json
{
  "protocol": 2,
  "type": "inventory",
  "device_id": "dev_xxx",
  "accounts": [
    {
      "account_uid": "uuid",
      "enabled": true,
      "healthy": true,
      "proxy_ready": true,
      "active_requests": 0,
      "max_concurrency": 3
    }
  ]
}
```

只上报路由元数据，不上报：

- Access Token。
- Refresh Token。
- API Key。
- 代理密码。
- Cookie。
- 完整错误堆栈。

库存数据保存在对应 Owner Relay Durable Object 的连接内存中。普通心跳不应每 20 秒写 D1。

以下事件触发库存更新：

- Relay 连接成功。
- 账号导入、删除、启用或关闭。
- 账号代理发生变化。
- 健康检查状态变化。
- 并发从 0 变为非 0 或达到上限。
- 工作区切换。

### 5.4 Worker 选路算法

收到共享请求后：

1. 根据 Guest Key 找到 grant 和 share group。
2. 根据账号池策略选择目标 `account_uid`。
3. 读取该账号配置的主设备和显式备用设备。
4. 过滤离线、无账号、账号关闭、代理不可用和并发已满设备。
5. 优先选择固定主设备。
6. 主设备不合格且配置了备用设备时，按明确优先级选择。
7. 没有合格设备时尝试共享池下一个账号。
8. 所有账号均无设备时返回可操作错误。

禁止使用：

```text
任意在线设备
-> 主设备布尔值排序
-> 取第一台
```

### 5.5 请求生命周期和禁止重复

协议必须区分：

```text
relay_request
relay_accepted
upstream_started
response_start
response_chunk
response_end
relay_error
```

故障转移规则：

| 失败时机 | 是否可切换设备 | 处理 |
| --- | --- | --- |
| 未找到设备 | 可以 | 尝试备用设备或下一个账号 |
| 设备拒绝，尚未开始上游 | 可以 | 生成新的 attempt_id 后重试 |
| WebSocket 在 upstream_started 前断开 | 可以 | 取消旧 attempt 后重试 |
| 已收到 upstream_started | 不可以 | 返回 `relay_result_unknown` |
| 已发送 response_start | 不可以 | 终止流并记录不完整结果 |

每次重试使用相同业务 `request_id` 和不同 `attempt_id`。Worker 必须保证同一个 request 只有一个 attempt 能越过 `upstream_started` 边界。

### 5.6 多设备界面

设备页展示：

- 设备名称。
- 在线/离线。
- 当前是否为本机。
- 本机可用账号数。
- 正在承载的共享数。
- 当前活跃请求数。
- 最近心跳。
- 设为新共享默认设备。
- 移除设备。

“默认设备”只影响以后创建的共享，不得静默改变已有共享的路由。

共享详情展示：

```text
承载设备：办公室电脑（在线）
备用设备：家里电脑（在线）
当前回流：办公室电脑
```

用户切换工作区前，如果本机承载共享，应显示：

```text
切换后，本机承载的 3 个共享会离线。
确认切换 / 取消
```

## 6. 移除好友功能并简化共享

### 6.1 从产品界面移除

移除：

- 好友标签页。
- 好友申请列表。
- Friend Code 展示与复制。
- 添加好友表单。
- 共享前选择好友。
- 等待好友接受后才能共享。

保留的概念：

- 共享码。
- 临时密码。
- 已授权用户。
- 收到的共享。
- 最近共享。
- 暂停、删除、限流和额度。

### 6.2 发起共享

首次使用：

```text
点击“发起共享”
-> 选择一个或多个账号
-> 确认由“此电脑”承载
-> 点击“开始共享”
-> 获得共享码和临时密码
```

再次使用：

```text
点击“开始共享”
-> Amber 复用上次账号池
-> 生成新临时密码
```

默认安全值：

- 密码有效期 10 分钟。
- 默认允许 1 位用户领取。
- 领取成功后该密码立即失效。
- 已领取的授权持续存在，直到共享者撤销或到期。
- 不默认共享全部账号。

“允许多人领取”放在高级设置中，可配置人数和有效时间。

### 6.3 接收共享

```text
输入共享码
-> 输入临时密码
-> 点击“连接”
-> Worker 创建该接收者独立 grant
-> Amber 保存独立 Guest Key
-> 自动加入收到的共享
-> 自动执行真实路径测试
-> 成功后出现在账户页面
```

领取过程不创建好友关系，也不等待共享者再次确认。共享者通过临时密码的发放行为完成授权确认。

### 6.4 授权管理

每位接收者单独显示：

- 脱敏邮箱或显示名称。
- 授权时间。
- 最近使用时间。
- 今日请求数。
- RPM。
- 并发限制。
- 总请求额度。
- 到期时间。
- 暂停、恢复、删除。

产品文案统一使用“接收者”或“已授权用户”，不使用“好友”和“联系人”。

### 6.5 旧好友数据迁移

1. D1 好友表在 v0.4.4 不直接删除。
2. 前端停止请求好友和好友申请接口。
3. 旧 share group、grant、Guest Key 保持有效。
4. 已有好友型共享的接收者转为普通授权记录。
5. 未处理好友申请不自动生成授权。
6. 旧 API 暂时只为升级兼容保留，记录弃用日志。
7. 确认两个正式版本无人依赖后，再单独安排数据库清理，不在 v0.4.4 删除历史表。

## 7. 共享池故障隔离

共享池中的账号失败不能拖垮整组共享。

### 7.1 失败分类

| 类型 | 处理 |
| --- | --- |
| DNS、连接超时、代理连接失败 | 本设备临时冷却，继续下一个账号 |
| 本设备不存在账号 | 尝试该账号的显式备用设备 |
| 401/403 凭据失效 | 二次确认后标记凭据异常 |
| 429 | 按响应头设置限流恢复时间 |
| 上游 5xx 且未开始流 | 尝试下一个账号 |
| 已开始上游或响应流 | 不重放，记录结果未知或流中断 |

设备本地网络失败不得同步成全局永久禁用。只有确定的凭据失效或用户手动关闭才属于可同步账号状态。

### 7.2 账号网络语义

每个账号明确保存：

- 强制直连。
- 跟随系统。
- 指定代理。

旧版无 `proxy_id` 账号迁移为“强制直连”，已有代理账号迁移为“指定代理”。

检测到 Clash Fake-IP 地址时提供明确诊断，不把 Fake-IP 连接失败误报成账号失效。

## 8. Codex 注入闭环修复

当前“账户测试成功但服务未启动，注入后 Codex 返回 502”的问题必须在 v0.4.4 消除。

将“注入配置”改为：

```text
启动服务并注入
```

执行顺序：

1. 检查本地 API Key 已配置。
2. 启动本地 API Server。
3. 如果端口占用，禁止继续并显示端口和占用信息。
4. 请求 `http://127.0.0.1:<port>/health`。
5. 验证响应包含 `service=amber-gateway` 和当前实例标识。
6. 使用本地 API Key 请求 `/v1/models`。
7. 两项验证通过后才写 `config.toml` 和 `auth.json`。
8. 写入后重新读取并校验 Base URL、API Key 指纹和模型。
9. 页面显示“服务运行中，Codex 配置有效”。

不得在服务未运行时仅根据设置中的端口生成配置。

WSL、Dev Container、SSH Remote 场景必须引导使用对应远程注入和隧道，不得把 Windows 的 `127.0.0.1` 当成远程环境地址。

## 14. 必须通过的测试矩阵

### 14.1 同机多用户

| 场景 | 预期结果 |
| --- | --- |
| A 登录、同步、退出、重新登录 A | 数据完整且不重复 |
| A 退出后尝试在 A 工作区登录 B | 保存会话前拒绝 |
| 创建 B 工作区并登录 B | 看不到 A 的账号、代理、共享和日志 |
| A 有待发送 outbox 时切换到 B | A 的 outbox 不会发送 |
| B 修改账号后切回 A | A 数据不发生变化 |
| 远端注销失败 | 本地仍能安全退出，工作区归属保留 |
| 切换过程中 Sidecar 崩溃 | 下次启动恢复到一个明确工作区 |
| 旧数据库检测到多个 user_id | 进入恢复工作区，停止自动同步 |
| 旧数据库存在归属不明 outbox | 隔离批次，不上传 |

### 14.2 同账号多设备

| 场景 | 预期结果 |
| --- | --- |
| A、B 登录同账号，共享由 A 创建 | 请求固定走 A |
| B 被设为“默认设备” | 已有共享仍走 A |
| A 在线但没有目标账号 | 不应向 A 发起该账号请求 |
| A 离线且未配置备用 | 返回共享设备离线 |
| A 离线，B 是显式备用且健康 | 上游开始前切到 B |
| B 有账号但代理不可用 | B 不进入候选集 |
| A 在 upstream_started 后断线 | 不切 B，返回结果未知 |
| A 工作区切换到其他云用户 | A 的原共享立即显示离线 |
| Worker-direct API 账号 | 不依赖 A/B 本地回流 |

### 14.3 共享和好友迁移

- 不添加好友即可领取共享。
- 单人临时密码领取后失效。
- 多人模式按人数限制。
- 每个接收者 Guest Key 不同。
- 暂停一人不影响其他人。
- 旧分享组和旧 Guest Key 继续有效。
- 旧好友记录不显示但不删除。

### 14.4 Codex 注入

- 服务停止时点击按钮会先启动服务。
- 服务启动失败时不写配置。
- 8080 被占用时显示明确错误。
- 健康接口不是 Amber 时拒绝注入。
- 注入后 `/v1/models` 验证成功。
- WSL/SSH 场景不会误用本地注入。
- 覆盖安装后配置仍指向实际运行端口。

### 14.5 常规门禁

- Go 全量测试。
- Go race 测试。
- Worker 测试。
- Vue/Vitest 测试。
- TypeScript 类型检查。
- Vite 生产构建。
- Rust 测试和格式检查。

## 16. 发布验收标准

只有同时满足以下条件才允许发布 v0.4.4：

1. 能以自动化测试证明 A 用户数据不会进入 B 用户工作区或云保险库。
2. 能以自动化测试证明同账号多设备共享固定走创建共享的设备。
3. 未显式配置备用设备时，Worker 不会随机换设备。
4. 上游开始后不存在跨设备重复请求。
5. 好友不是共享的前置条件，普通流程不出现好友页面。
6. 已有共享和 Guest Key 升级后仍然可用。
7. 服务未启动或端口冲突时不再生成无效 Codex 配置。
8. v0.4.3 数据迁移过程不删除原数据。
9. 官网、文档、Release 和应用内版本全部一致。
10. 只生成和发布一个 v0.4.4 正式安装包。

## 17. 明确不做的事情

- 上游请求开始后的自动重放。
- 在官网或 Worker 中托管 EXE 安装包。
- Cloudflare 容量和成本扩容方案。
