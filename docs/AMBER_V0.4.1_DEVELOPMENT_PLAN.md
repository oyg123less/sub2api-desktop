## 1. 版本定位

v0.4.1 定位为 **可靠性与高频操作体验修复版本**。本版本不继续扩张云端共享业务范围，优先解决当前已经影响使用的云同步连接失败，并完成账户管理、费用统计和导航层级的整理。

本版本交付目标如下：

1. 云同步失败时，用户能看懂失败发生在 DNS、连接、TLS、HTTP 还是登录会话阶段，并能在 Amber 内选择合适的网络路径后直接重试。
2. 账户行在默认窗口尺寸下保持清晰，不再把并发/排队单独挤占一列；额度消耗有直观进度条。
3. 单个删除入口移到账户行最右侧，并支持可确认、可审计、事务安全的批量删除。
4. 代理页面可以把指定代理一次性绑定到当前全部账户，也可以一键清空全部账户代理。
5. 账户页面可以对所选账户或全部账户发起受控并发测试，并持续显示进度和逐账户结果。
6. 仪表盘展示今日预估费用，统计页展示所选周期及各模型预估费用。
7. 设置入口固定在左侧栏底部，局域网访问只在仪表盘保留一个控制入口。
8. 将现有邮件可靠性热修纳入 v0.4.1 正式基线，避免后续开发或合并时丢失。

## 3. 当前代码与问题分析

### 3.1 账户行现状

当前 `src/views/Accounts.vue` 的账户行使用六段网格：身份、健康状态、Token、预估费用、并发/排队、操作。默认窗口下信息仍能显示，但横向扩展后产生以下问题：

- 并发/排队属于账户身份与调度状态的一部分，却单独占用一列，视觉关系较弱。
- OAuth 额度窗口只在详情弹窗展示，列表无法快速判断哪个账户接近限额。
- Token 与预估费用分别占列，账户较多时扫读效率低。
- 删除按钮只在详情弹窗底部，删除单个账户需要先打开详情并滚动。
- 当前没有选择状态、全选入口或批量删除接口。

现有可复用数据：

- `Account.in_flight`、`max_concurrency` 可直接显示并发占用。
- `Account.waiting`、`queue_capacity` 可直接显示排队占用。
- `Account.codex_usage` 已包含主要和次要额度窗口、使用百分比及重置时间。
- `GET /control/accounts` 已返回每个账户的 Token 与 `cost_usd` 汇总。

### 3.2 删除链路现状

当前只有以下单删链路：

```text
DELETE /control/accounts/{id}
  -> Control.deleteAccount
  -> Store.DeleteAccount
  -> DELETE FROM accounts WHERE id=?
```

数据库已有 `accounts_cloud_delete` 触发器。每个账户被删除前都会生成对应的云同步 tombstone，因此批量删除必须保留逐账户触发器行为。

不能采用“前端循环调用单删接口”的方案。该方案在第 N 个请求失败时会形成半完成状态，用户无法判断哪些已经删除，而且会产生多次刷新、多个确认流程和更复杂的云同步状态。

正确方案是在后端新增一个批量接口，由 Store 在 **单个 SQLite 事务** 中处理合法 ID、缺失 ID和删除操作。任何数据库错误都回滚整批删除；成功提交时，每个被删除账户都必须保留一条 tombstone。

### 3.3 代理批量绑定现状

当前代理页面只支持代理的添加、编辑、测试和删除。账户绑定代理只能进入单个账户详情后调用：

```text
POST /control/accounts/{id}/proxy
  -> Store.SetAccountProxy(id, proxyID)
```

当账户数量较多时，逐账户绑定无法使用。前端循环调用单账户接口仍会产生半完成状态：中途失败后，一部分账户已经切换代理，另一部分仍是旧配置，而且每次更新都会独立触发云同步 dirty 状态。

v0.4.1 应增加后端事务批量绑定能力，并从代理页提供以下操作：

- 将某个代理应用到当前全部账户。
- 选择“全部直连”以清空当前全部账户的 `proxy_id`。
- 显示当前账户代理摘要：全部直连、全部使用同一代理或混合配置。
- 显示每个代理当前绑定的账户数量。

“全局选择”在本版本中定义为 **对当前全部账户执行一次批量写入**，不是运行时优先级更高的永久全局覆盖层。批量应用后仍可单独修改某个账户；以后导入的新账户继续使用导入流程中选择的代理，不会被隐藏规则自动覆盖。

### 3.4 账户批量测试现状

当前 `POST /control/accounts/{id}/test` 是最长 100 秒的同步请求，并通过完整 TLS 指纹、代理、OAuth 刷新和真实上游请求链路测试账户。它会产生一条最小真实请求、写入用量日志，并根据结果更新临时限额、认证失败或成功状态。

直接由前端同时循环测试几十个账户存在以下风险：

- WebView 会持有大量长时间 HTTP 请求，页面关闭或刷新后难以恢复进度。
- 全部账户使用同一代理时可能瞬间压满代理连接。
- 大量同时请求可能触发上游速率限制，导致测试结果失真。
- 一个请求异常不应中断其他账户，但前端难以统一处理取消、超时和结果汇总。
- 测试会消耗少量 Token 并产生预估费用，必须在开始前明确告知用户。

因此批量测试需要后端异步任务和有上限的 worker pool，不能实现为 `Promise.all(accounts.map(testAccount))`。

### 3.5 预估费用现状与隐藏风险

当前账户列表已有预估费用，后端通过 `openai.CostUSD()` 和官方价格目录计算；价格版本与来源位于 `core/internal/openai/pricing.go`：

- 价格版本：`openai.PriceVersion`
- 官方来源：`openai.PriceSourceURL`
- 价格维度：输入、缓存输入、输出、长上下文档

但统计接口当前只返回请求数、Token 与延迟：

- `StatsSummary` 没有 `cost_usd`。
- `DailyPoint` 没有 `cost_usd`。
- `ModelPoint` 没有 `cost_usd`。
- 前端 `StatsResponse` 也没有费用和价格元数据。

现有账户费用还有一个必须同时修正的精度风险：`UsageByAccountModel()` 先把同一账户、同一模型的多次请求 Token 合并，再调用 `CostUSD()`。如果许多短请求的累计输入超过 272K，汇总值可能被错误地按“单次长上下文请求”计价。

v0.4.1 必须改为 **先按每次请求判断长上下文档并计算费用，再按账户/日期/模型汇总**。测试必须覆盖“很多短请求累计超过 272K，但任何单次请求都未超过 272K”的场景。

### 3.6 设置入口与局域网开关现状

当前 `src/App.vue` 把设置放在主导航中；侧栏底部已有后端状态、主题和版本区域。设置更适合作为低频全局入口固定在侧栏底部，避免和账户、代理、统计等业务页面处于相同层级。

`allow_lan` 同时出现在仪表盘和设置页。仪表盘已有确认弹窗和服务重启行为，应作为唯一控制入口。设置页移除开关后仍需保留从后端读取到的 `allow_lan` 原值，保存其他设置时不得把它误写为 `false`。

### 3.7 云同步失败根因

2026-07-18 的本机诊断结果表明，当前失败发生在访问 Worker 之前：

| 检查项 | 结果 |
| --- | --- |
| Sidecar 实际 Cloud URL | `https://amber-cloud-api.484486528.workers.dev`，配置正确 |
| Worker 状态 | 此前已验证可返回 HTTP 200，服务本身可用 |
| 当前 IPv4 解析 | `69.50.221.20`，连接在 TCP/TLS 前超时 |
| 当前 IPv6 解析 | `2a03:2880:f11f:83:face:b00c:0:25de`，解析/连接失败 |
| 环境代理 | 未发现可用的 `HTTP_PROXY`、`HTTPS_PROXY`、`ALL_PROXY` |
| WinHTTP | 直连 |
| Amber 连续失败 | 已达到 20 次，后台重试仍失败 |

上述地址与预期的 Cloudflare Worker 网络路径不一致，现象符合本地 DNS 污染、错误解析或当前网络无法访问 `workers.dev`。失败发生在鉴权、拉取、推送之前，因此重新登录或继续增加同步重试次数不能解决根因。

当前后端已经具备的能力：

- 默认 Cloud HTTP Client 会依次读取环境代理和 Windows Internet Settings 代理。
- 单次 GET/HEAD 和幂等 PUT 有 500ms、1500ms 短重试及随机抖动。
- Manager 对可重试错误按 5 秒、15 秒、1 分钟、5 分钟进行后台退避。
- 错误已经能标记为 `dns`、`connect`、`tls`、`http`、`response`、`timeout`、`local`。
- Cloud 页面已经显示最近失败、连续失败次数、下次重试时间和手动重试。

当前缺失的闭环：

- 用户无法在 Amber 内明确选择“系统代理、Amber 已保存代理、强制直连”。
- 页面不显示当前实际采用的代理来源。
- 没有独立的 DNS、TCP、TLS、`GET /health` 分阶段探测结果。
- 修改网络路径后不能一键“应用并立即重试同步”。
- 当前提示虽然有错误阶段，但对普通用户仍缺少明确的处理动作。

结论：v0.4.1 的正确修复是补全 **网络路径选择 + 分阶段诊断 + 实时重建 Cloud Client + 立即重试**，而不是固定 IP、无限重试或清空登录会话。

## 4. 账户页面设计

### 4.1 桌面端账户行

默认宽度下采用以下信息层级：

```text
┌────────────────────────────────────────────────────────────────────────────────────────────────────┐
│ [□] [头像] 账户名称                 [健康状态]      [额度消耗]              [累计用量]    [开关][测][详][删] │
│          [类型] 套餐/Base URL       最近成功/原因    5h  █████░ 64%          106.73M                    │
│          并发 1/3 · 排队 0/20                       7d  ███░░░ 31%          约 $609.8308               │
└────────────────────────────────────────────────────────────────────────────────────────────────────┘
```

布局规则：

1. 最左侧为选择框，始终可见，避免只有 hover 才能发现批处理功能。
2. 账户名称下依次放账户类型/套餐和“并发 x/y · 排队 x/y”。并发与排队不再占独立列。
3. 健康状态保持独立，严重状态原因继续单行截断并提供完整 `title`。
4. 额度区最多显示两个迷你进度条，对应现有主要/次要窗口；显示窗口长度、使用百分比和重置时间。
5. 累计 Token 与预估费用合并到同一紧凑指标区，避免各占一列。
6. 最右操作区固定顺序：启用开关、测试、查看详情、删除。
7. 删除使用垃圾桶图标按钮和 tooltip，保留危险色 hover/focus 状态，不使用大面积红色常驻按钮。
8. 详情弹窗移除删除按钮，避免两个删除入口产生行为差异。

额度显示规则：

- `used < 70%` 使用正常色。
- `70% <= used < 90%` 使用警告色。
- `used >= 90%` 使用危险色。
- OAuth 账户有两个额度窗口时按窗口长度从短到长展示。
- 仅有一个窗口时只显示一个进度条，不保留空行。
- API Key 或缺少 `codex_usage` 的账户显示“额度数据不可用”，不能伪装为 0%。
- 百分比限制在 0 到 100 的视觉范围内，但原始异常数据应在详情或诊断日志中保留。

### 4.2 响应式行为

| 容器宽度 | 行为 |
| --- | --- |
| `> 1180px` | 完整展示身份、健康、双额度条、累计用量和行内操作 |
| `900px - 1180px` | 操作按钮压缩为图标；隐藏额度重置文字，保留百分比和进度条 |
| `720px - 900px` | 额度区移动到账户身份下方第二行；累计用量保持一组 |
| `< 720px` | 单列信息流；操作固定在行底部；不允许横向页面溢出 |

必须在 Tauri 默认尺寸 `1280 x 820`、最小尺寸 `900 x 600` 和宽屏截图尺寸验证。hover 的轻微缩放与阴影可保留，但选择框、标签或动态数据不得导致行高抖动。

### 4.3 多选与批量操作交互

选择状态使用 `Set<number>` 保存账户 ID，不直接保存 Account 对象，避免一秒一次的运行状态轮询替换对象后丢失选择。

交互流程：

```text
勾选一个或多个账户
  -> 列表上方出现上下文操作条
  -> 显示“已选择 N 个”、全选/取消全选、测试所选、删除所选
  -> 删除所选：确认影响 -> 后端事务删除 -> 刷新账户和待同步数量
  -> 测试所选：选择模型 -> 确认实际请求 -> 后端异步测试任务 -> 持续显示进度
```

选择规则：

- 表头复选框支持未选、部分选择、全选三态。
- “全选”默认选择当前已加载的全部账户；当前页面没有分页，因此语义明确。
- 账户运行状态轮询时，保留仍存在的选中 ID，自动移除已经不存在的 ID。
- 删除进行中锁定选择框、行内删除和批量删除按钮，防止重复提交。
- 有选中账户时，“批量测试”默认测试所选；没有选中账户时，页面标题区提供“测试全部账户”。
- 单删与批量删除共用同一确认文案体系和同一后端事务实现。
- 成功提示分别显示已删除数量和已不存在数量。
- 失败时后端回滚，前端不得先乐观移除行。

确认文案必须明确：删除会移除本机凭据；登录云账户时，删除记录将在后续同步到其他设备。弹窗不得展示 access token、refresh token 或 API Key。

## 5. 账户批处理接口与安全规则

### 5.1 Control API

新增接口：

```http
POST /control/accounts/batch-delete
Content-Type: application/json

{
  "account_ids": [12, 18, 25]
}
```

成功响应：

```json
{
  "ok": true,
  "requested": 3,
  "deleted": [12, 18],
  "missing": [25],
  "failed": []
}
```

事务失败响应：

```json
{
  "error": {
    "code": "batch_delete_failed",
    "message": "The account batch could not be deleted.",
    "retryable": true,
    "details": {
      "requested": 3,
      "deleted": [],
      "failed": [12, 18, 25]
    }
  }
}
```

接口规则：

- `account_ids` 必须是数组，长度范围为 1 到 500。
- 每个 ID 必须为正整数。
- 后端去重，不依赖前端去重。
- 不存在的 ID 放入 `missing`，不阻止其他有效 ID 删除。
- 任意数据库执行或提交错误必须回滚；失败响应的 `deleted` 必须为空。
- 保留现有 `DELETE /control/accounts/{id}`，并让其复用新的 Store 删除实现，确保旧前端和内部调用兼容。

### 5.2 Store 事务

新增建议方法：

```go
func (s *Store) DeleteAccounts(ids []int64) (BatchDeleteResult, error)
```

事务顺序：

1. 对输入 ID 做正整数校验、去重和稳定排序。
2. 开启 SQLite 事务。
3. 查询实际存在的 ID，生成 `deleted` 和 `missing` 候选集合。
4. 对实际存在的 ID 执行参数化删除。
5. 验证受影响行数与候选数量一致。
6. 提交事务。
7. 仅在提交成功后返回已删除结果。

现有 `accounts_cloud_delete` 为 `BEFORE DELETE` 触发器，因此同一事务内会为每个账户创建 tombstone；如果事务回滚，账户删除和 tombstone 也必须一起回滚。

### 5.3 全部账户代理配置

代理页面在列表上方增加“账户代理配置”工具区：

```text

│ 账户代理配置   当前：混合配置   已绑定 84 / 99                      │
```

每个代理行同时显示“已绑定 N 个账户”，并在行操作菜单提供“应用到全部账户”。危险性较低但影响面较大，点击后仍需要一个简短确认弹窗，显示目标代理、受影响账户数和将被覆盖的现有绑定数。

应用成功结果区提供“测试全部账户”后续动作，形成清晰流程：

```text
选择代理 -> 应用到全部账户 -> 查看 updated/unchanged -> 测试全部账户
```

状态摘要规则：

- 全部 `proxy_id=NULL`：显示“全部直连”。
- 所有账户绑定同一代理：显示代理名称和“全部 N 个账户”。
- 存在多个代理或部分直连：显示“混合配置”，并展示已绑定/总数。
- 没有账户：禁用应用按钮并显示“暂无账户”。

新增接口：

```http
GET  /control/accounts/proxy-summary
POST /control/accounts/batch-proxy
```

绑定全部账户请求：

```json
{
  "scope": "all",
  "proxy_id": 7
}
```

取消全部账户代理请求：

```json
{
  "scope": "all",
  "proxy_id": null
}
```

成功响应：

```json
{
  "ok": true,
  "matched": 99,
  "updated": 84,
  "unchanged": 15,
  "proxy_id": 7
}
```

后端规则：

- `scope` v0.4.1 只接受明确的 `all`，不以空 ID 数组暗示全部账户。
- 非空 `proxy_id` 必须先验证代理存在。
- 账户查询、批量更新和结果计数必须在一个 SQLite 事务内完成。
- SQL 只更新代理实际发生变化的账户，避免无意义地刷新 `updated_at` 和云同步版本。
- 现有 `accounts_cloud_dirty` 触发器必须为每个发生变化的账户标记 `sync_dirty=1`，使代理绑定同步到用户其他设备。
- 任意执行或提交错误都回滚全部账户，不允许部分切换。
- 操作只改变 `proxy_id`，不得改变账户启用状态、并发、队列、额度或凭据。
- 新导入账户不继承这次批量操作；导入时仍由现有代理选择器决定。
- 有批量账户测试正在运行时，批量代理切换返回 409 并提示先结束测试，避免同一批结果混用切换前后的代理。

建议 Store 方法：

```go
func (s *Store) SetAllAccountsProxy(proxyID *int64) (BatchProxyResult, error)
```

任务执行规则：

- 单次最多 500 个账户，输入 ID 去重；任务开始时只保存稳定的账户 ID 顺序和脱敏标签，不长期复制整批凭据。
- worker 开始测试某个 ID 前重新读取最新账户与代理绑定；账户已被删除时标记为 `skipped_account_missing`，不得继续使用旧凭据发请求。
- 全局 worker 并发固定为 3；同一代理同时最多运行 2 个测试，避免一条代理链路被瞬间压满。
- 每个账户继续走现有 `Engine.TestAccount()` 完整链路，并保留单账户超时上限。
- 每个账户失败只记录该项结果，不中断整批任务。
- 同一时刻只允许一个批量测试任务；重复启动返回 409 和当前 `run_id`。
- 批量代理事务正在执行时不能创建测试任务；代理应用完成后才能开始测试。
- `DELETE` 会取消尚未开始的账户并取消正在执行的 context；已经完成的结果保留。
- 任务完成后结果在内存保留 30 分钟，页面刷新或重新进入账户页可通过 `active`/`runId` 恢复显示；Sidecar 重启后任务丢失是可接受行为。
- 批量响应不保存模型输出 sample，只保留脱敏账户标签、状态、延迟、错误类别和截断后的错误信息。
- 为现有 `gateway.TestResult` 增加向后兼容的结构化 `error_kind`，批量页面不得依赖解析中文/英文错误字符串进行分类。
- 成功测试必须继续保留用户手动禁用状态；认证失败、限额和网络错误沿用现有单账户测试状态规则。
- 测试会发送真实最小请求并写入请求日志，开始前必须提示可能消耗少量 Token 和产生预估费用。

批量测试 UI 使用可最小化的进度面板：显示进度条、成功/失败/取消计数、当前运行数量、取消按钮，以及“全部、成功、失败”筛选。关闭面板不取消任务；重新打开账户页面应恢复活动任务。失败明细必须限制宽度并支持复制，不能让长错误文本撑出卡片。

## 6. 预估费用设计

### 6.1 计算原则

所有费用均在 Go 后端计算，前端只负责格式化和展示。

费用计算顺序必须是：

```text
单条请求日志
  -> 根据实际模型匹配价格
  -> 根据该请求 prompt_tokens 判断是否进入长上下文档
  -> 拆分未缓存输入、缓存输入、输出
  -> 得到该请求 cost_usd
  -> 再按账户、日期、模型或统计周期求和
```

以下算法禁止使用：

```text
先把多条请求的 Token 合并
  -> 再用合计 prompt_tokens 判断是否超过 272K
```

未知模型继续使用当前 `PriceForModel()` 的回退规则，但响应要返回回退请求数量，使 UI 可以说明其中部分费用采用兼容价格估算。

## 8. 云同步网络修复设计

### 8.1 设备本地连接设置

新增设备本地的 `cloud_connection_settings`。v0.4.0 已使用 schema 12，因此本版本数据库 schema 从 12 升到 13：

```sql
CREATE TABLE cloud_connection_settings (
  id INTEGER PRIMARY KEY CHECK (id = 1),
  mode TEXT NOT NULL DEFAULT 'system'
    CHECK (mode IN ('system', 'proxy', 'direct')),
  proxy_id INTEGER NULL REFERENCES proxies(id) ON DELETE SET NULL,
  updated_at INTEGER NOT NULL
);
```

模式语义：

| 模式 | 行为 |
| --- | --- |
| `system` | 保持当前默认逻辑：环境代理优先，其次 Windows Internet Settings，最后直连 |
| `proxy` | 使用 Amber 代理页面中选定的 HTTP、HTTPS 或 SOCKS5 代理 |
| `direct` | 明确禁用环境代理和系统代理，强制直连 |

该表不添加云同步 dirty trigger，不进入 `settingsPayload`，不上传 Worker。代理密码继续使用现有代理表的加密存储，不新增明文 URL 或凭据字段。

如果 `mode=proxy` 但代理已被删除，返回 `cloud_proxy_missing`，不得悄悄回退到直连。用户必须明确选择系统代理、另一个代理或直连。

### 8.2 本地 Control API

新增接口：

```http

```

读取响应示例：

```json

  "effective_source": "windows",
  "effective_proxy_type": "http",
  "last_probe": null

```

保存请求示例：

```json

  "proxy_id": 4
}
```

探测响应示例：

```json
{
  "ok": false,
  "target": "amber-cloud-api.484486528.workers.dev",
  "effective_source": "amber_proxy",
  "proxy_name": "Local SOCKS5",
  "stages": [
    { "id": "dns", "status": "skipped", "detail": "Target DNS is delegated to the proxy." },
    { "id": "connect", "status": "ok", "latency_ms": 42 },
    { "id": "tls", "status": "ok", "latency_ms": 81 },
    { "id": "http", "status": "failed", "http_status": 502, "latency_ms": 113 }
  ],
  "error_code": "cloud_http_502",
  "error_stage": "http"

```

安全要求：

- API 和日志只返回代理来源、代理名称和类型，不返回主机凭据、密码或带认证信息的 URL。
- 探测只访问配置的 Amber Cloud 主机和 `/health`，不接受前端传入任意目标 URL，避免形成 SSRF 工具。
- 每个阶段有独立超时，总探测时间有硬上限，并响应请求取消。
- 网络错误日志记录错误类型和阶段，不记录 access token、refresh token、验证码或请求正文。

### 8.3 Manager 实时重配置

修改网络模式后不要求重启 Amber：

1. 校验模式和代理 ID。
2. 持久化设备本地连接设置。
3. 在 Manager 操作锁下构建新的 Cloud HTTP Client。
4. 原子替换 Client，避免后台同步与手动探测发生数据竞争。
5. 清除旧网络错误的下次重试计时，但不清空登录会话、同步 cursor 或 outbox。
6. 用户点击“应用并重试”时立即执行一次同步。

短重试和后台退避保持现状。网络路径切换后从第一次失败计数重新开始；普通后台失败不得自动注销用户。只有明确的终止会话错误（如无效 refresh token、会话过期、账户禁用）才能清除本地云会话。

### 8.4 Cloud 页面交互

同步失败卡增加两个动作：

- “网络诊断”：运行分阶段探测并展示 DNS、连接、TLS、HTTP 四阶段结果。
- “连接设置”：选择使用系统代理、Amber 已保存代理或直连。

连接设置面板底部提供“应用并重试”。成功时关闭错误态并刷新工作区；失败时保留诊断结果和当前选择，不能只弹一条瞬时 toast。

面向普通用户的错误提示：

| 阶段 | 中文提示 | 建议动作 |
| --- | --- | --- |
| DNS | 无法正确解析 Amber Cloud | 切换系统 DNS 或选择可用代理 |
| Connect | 无法建立网络连接 | 检查代理、网络或防火墙 |
| TLS | 安全连接验证失败 | 检查系统时间、证书拦截和代理 |
| HTTP | Amber Cloud 返回错误 | 保留状态码并稍后重试 |
| Timeout | 连接 Amber Cloud 超时 | 选择可用代理或更换网络 |
| Session | 云登录已失效 | 仅此时提示重新登录 |
| Local | 本地数据或配置处理失败 | 展示可复制诊断信息 |

“当前网络路径”只显示类似“系统代理”“Amber 代理：Local SOCKS5”“直连”，不得显示敏感代理 URL。

### 8.5 自定义 Cloud 域名说明

自定义 Amber API 域名可以作为后续部署优化，但不作为本次故障的临时补丁。当前 `amberapp.asia` 使用阿里云 DNS；Cloudflare Worker Custom Domain 通常要求域名或对应 DNS 区域由 Cloudflare 管理。除非后续明确迁移 DNS 或增加受支持的 Cloudflare 路由，否则 v0.4.1 继续使用已部署 Worker URL。

- 代理页显示全部账户当前是全部直连、同一代理还是混合配置。

## 15. 风险与控制

| 风险 | 控制措施 |
| --- | --- |
| 批删中途失败导致部分账户消失 | 单事务、提交后才更新 UI、故障注入测试 |
| tombstone 缺失导致其他设备复活账户 | 保留现有删除触发器并验证每账户一条 tombstone |
| 全部账户代理切换中途失败 | 单事务、只更新变化行、提交后刷新摘要 |
| “全局代理”与单账户代理优先级冲突 | 定义为一次性批量写入，不新增永久覆盖层 |
| 批量测试打满同一代理或触发上游限流 | 全局并发 3、同代理并发 2、单任务限制和取消 |
| 批量测试被误认为免费本地检查 | 启动前说明真实请求、Token、费用和状态副作用 |
| 长错误或 500 条结果占用过多内存/UI | 错误截断、结果分页/虚拟化、30 分钟 TTL |
| 额度条让行过高 | 最多两条迷你进度、稳定高度、窄屏降级 |
| 费用因长上下文判断顺序而失真 | 每请求计价后汇总，加入累计阈值反例测试 |
| 未知模型费用被误认为精确值 | fallback 计数与“预估”说明 |
| 代理凭据泄露 | 复用加密代理记录，API/日志只返回脱敏来源 |
| 代理配置同步后使其他设备失联 | 连接设置严格设备本地化，不进入云 payload |
| 热切换 Client 与后台同步竞争 | Manager 操作锁、原子替换、race/并发测试 |
| 设置页移除 LAN 后保存覆盖状态 | 保留字段原值并做 E2E 回归 |
| 开发或打包干扰用户正在运行的软件 | 禁止安装、卸载、停止、重启；只产出并汇报文件 |
