# Amber v0.3.3 幂等与超时安全开发文档

> 目标版本：v0.3.3
> 基线版本：v0.3.2
> 范围：M3 云同步幂等、分页检查点、共享配额事务化
> 不包含：OAuth 设备回源 WebSocket 隧道（保留到 v0.4.0）

## 1. 目标

v0.3.3 解决请求已经在服务端完成、但客户端因超时或断线未收到响应时的数据一致性问题：

1. 同一批同步写入只能生效一次，重试返回第一次结果。
2. 客户端重启后仍能继续未确认批次，不产生重复版本或假冲突。
3. 拉取多页数据时逐页保存游标，后续页面失败不重做全部页面。
4. 共享请求先预占配额，成功才结算，失败必须释放。
5. 所有恢复路径均可通过故障注入测试验证。

## 2. 版本边界

### 2.1 本版本包含

- `PUT /v1/vault/batch` 支持 `Idempotency-Key`。
- D1 保存七天批次回执，并校验幂等键与请求哈希的绑定关系。
- D1 使用基础版本写入声明，阻止两个设备同时写入同一版本。
- 桌面端持久化同步 outbox，响应丢失或进程退出后复用原幂等键。
- 同步拉取每成功应用一页就保存复合游标检查点。
- 本地确认服务端回执时检查本地版本和更新时间，不覆盖回执期间产生的新修改。
- 共享配额增加 `pending -> settled/released` 预占状态机。
- 过期批次回执、写入声明和配额预占记录自动清理。

### 2.2 本版本不包含

- 不实现 OAuth 账号的设备回源共享。
- 不上传拥有者设备的 TLS 指纹、代理环境或 OAuth Token。
- 不改变现有 vault AES-256-GCM 密文格式和主密码派生方式。
- 不允许 Worker 自动重放上游模型请求。

## 3. 云同步协议

### 3.1 本地 outbox

本地数据库 Schema 11 新增 `cloud_sync_outbox`：

| 字段 | 用途 |
| --- | --- |
| `idempotency_key` | 批次唯一键，跨进程重试保持不变 |
| `payload_json` | 已完成端到端加密的批次载荷 |
| `created_at` | 排序和诊断时间 |

客户端发送前先写入 outbox。只有在收到并应用服务端回执后才删除；因此以下故障均可恢复：

- Worker 写入成功后连接断开。
- 客户端读取响应超时。
- Amber 在回执确认前退出。
- GET/PUT 重试耗尽后由后台同步再次接管。

### 3.2 服务端批次回执

D1 Schema 3 新增 `vault_batch_receipts`。主键为 `(user_id, idempotency_key)`，保存请求哈希、HTTP 状态和完整 JSON 结果。

- 同键同载荷：直接返回原结果，并设置 `Idempotency-Replayed: true`。
- 同键不同载荷：返回 `409 idempotency_key_reused`。
- 首次成功写入：vault 变更、账号共享撤销和成功回执放在同一个 D1 batch 中。
- 回执保留七天，过期后由请求路径异步清理。

### 3.3 并发版本声明

`vault_write_claims` 使用 `(user_id, kind, client_uid, base_version)` 唯一键。两个设备同时基于相同版本写入时只有一个批次可以提交；另一个重新读取当前版本并返回标准 `vault_conflict`，禁止出现“两个请求都成功但只有一个实际更新”的结果。

### 3.4 分页检查点

客户端使用 `updated_at|id` 复合游标。每页遵循固定顺序：

1. 拉取页面。
2. 解密并应用所有项目。
3. 持久化该页游标。
4. 继续下一页。

页面应用失败时不推进游标；下一页超时时保留上一页检查点。重复拉取由本地 `sync_version` 去重。

### 3.5 回执期间的本地修改

确认回执时同时比较 `client_uid`、基础版本和本地 `updated_at`：

- 本地未变化：更新云版本并清除 `sync_dirty`。
- 本地已有新修改：只推进基础云版本，保留 `sync_dirty=1`，下一批继续上传新内容。
- 本地项目已删除：将删除墓碑 rebase 到服务端新版本。

## 4. 共享配额状态机

D1 新增 `reserved_requests` 和 `share_request_reservations`。数据库触发器负责原子状态迁移：

```text
请求到达 -> pending（预占）
              |-- 上游 HTTP < 400 -> settled（used +1，reserved -1）
              |-- 网络异常/HTTP >= 400 -> released（reserved -1）
              `-- 超过 10 分钟未完成 -> 删除并自动释放
```

配额判断使用 `used_requests + reserved_requests < quota_requests`。成功、释放操作都带 `state='pending'` 条件，同一预占记录重复结算不会重复扣费。

## 5. 安全约束

- 幂等回执只保存端到端密文，不保存 vault key 或明文凭据。
- 日志禁止记录幂等载荷、Authorization、API Key、邮箱验证码和密文正文。
- 幂等键长度限制为 16 至 128 字符，只允许安全字符。
- 用户之间的幂等键通过 `user_id` 隔离。
- 共享失败响应继续过滤上游 HTML 和敏感响应头。

## 6. 故障注入与测试

必须覆盖：

- 同一幂等键重复 PUT，只增加一次版本。
- 同一幂等键绑定不同载荷，稳定返回 409。
- 不同幂等键并发写相同基础版本，一个成功、一个冲突。
- 服务端提交后丢弃响应，客户端重试和重启后取得原回执。
- 第二页拉取失败，第一页面游标仍已持久化。
- 回执到达前本地再次编辑，不得错误清除脏状态。
- 上游 5xx/网络失败释放共享配额，后续请求仍可使用。
- 上游成功只结算一次，预占计数归零。
- Go race、Worker Vitest、Vue Vitest、Playwright、Rust test/clippy 全部通过。

## 7. 部署顺序

1. 备份并迁移远端 D1 到 Schema 3。
2. 部署兼容旧客户端的 Worker；无幂等键的 v0.3.2 客户端仍可工作。
3. 验证 Worker `/health` 返回 `0.3.3`。
4. 构建 v0.3.3 sidecar 和桌面安装包。
5. 不运行安装包，不停止、安装或卸载用户当前 Amber。

回滚 Worker 时不得回滚或删除 D1 新表；旧代码会忽略新增字段和表。桌面端升级前自动备份本地数据库，回滚旧客户端时应使用升级前备份，避免旧版本拒绝较新的本地 Schema。

## 8. 完成定义

v0.3.3 只有在以下条件全部满足时完成：

1. 同步响应丢失不会重复写入或产生假冲突。
2. outbox 可跨 Amber 重启恢复。
3. 多页同步可从最后成功页面继续。
4. 共享失败不消耗配额，成功请求只结算一次。
5. D1 与本地 Schema 迁移测试通过。
6. 所有质量门禁通过并生成可校验的 v0.3.3 NSIS 安装包。
